Architecture · Section 03

Intake Workflow

Pre-submission phase — how a claimant's receipt image becomes a validated, persisted claim through the intake_gpt ReAct subgraph with 6 tools, multimodal extraction, RAG policy validation, and human-in-the-loop justification capture.

1

Intake Agent Tools

The 6 tools bound to the intake_gpt ReAct subgraph

Tool 1
getClaimSchema
Returns the structured field template the agent must populate before submitting. Anchors the ReAct loop to the required output schema.
Tool 2 · VLM
extractReceiptFields
Encodes the receipt image (base64), calls Gemini 2.0 Flash via OpenRouter, parses structured JSON response. OpenCV quality gate rejects blurry images before the VLM call.
Tool 3 · RAG
searchPolicies
Calls MCP-RAG → Qdrant. Semantic search over 5 policy documents (meals, transport, accommodation, office supplies, general). Returns top-N chunks with scores.
Tool 4 · FX
convertCurrency
Calls MCP-Currency → Frankfurter API. Converts any foreign currency to SGD. Returns original amount, converted amount, exchange rate, and date. Falls back to manual rate on API failure.
Tool 5 · DB
submitClaim
Calls MCP-DB → PostgreSQL. Creates claim + receipt row, writes initial audit log. Returns claimId + claimNumber. Sets claimSubmitted=true in state, triggering the evaluator gate.
Tool 6 · HIL
requestHumanInput
Triggers a LangGraph interrupt. Pauses the graph and surfaces a question to the user via SSE. Resumes when the user responds. Used for confirmations and justification capture.

Image store pattern

Receipt images are stored in an in-memory imageStore dict (keyed by claimId), not in the LangGraph state. This avoids embedding ~58 KB of base64 data into every checkpoint and LLM context.

# Web layer stores on upload
imageStore[claimId] = base64_image

# extractReceiptFields tool retrieves
image = imageStore.get(claimId)
→ passed to VLM directly
→ LLM context stays clean
